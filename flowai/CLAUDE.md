# CLAUDE.md — 문서 운영 규칙

본 문서는 플로우AI 기획 패키지의 문서를 작성·수정할 때 반드시 따라야 하는 규칙이다. 모든 문서 수정 작업은 본 규칙을 우선 적용한다.

---

## 0. 핵심 원칙 (3줄 요약)

1. **누적 금지** — 추가하지 말고 합쳐라. Append 대신 Rewrite.
2. **단일 정의 (SSOT)** — 같은 개념은 한 문서에서만 정의한다. 나머지는 참조.
3. **변경이력 분리** — 본문은 항상 "현재 상태"만. 이력은 CHANGELOG에만.

---

## 1. SSOT 매핑 — 어떤 정의가 어디에 있어야 하는가

본 패키지의 모든 핵심 개념은 아래 표의 "SSOT 위치"에서만 정의된다. 다른 문서는 절대 정의를 다시 쓰지 않고 참조 링크로만 처리한다.

| 개념 / 데이터 | SSOT 위치 | 다른 문서의 처리 |
|---|---|---|
| V1/V2/V3 스코프 결정·항목 수 | 01-PRD §4 | "V1 N개" 숫자 자체를 쓰지 않는다. "[01-PRD §4 참조]"만 |
| M1~W5 항목 식별자·라벨·소스 | 03-data-spec §1-5 | 식별자는 그대로 인용 가능. 라벨 재서술 금지 |
| 항목별 데이터 요구사항·필터 | 03-data-spec §3 | 참조만. 필드 목록 재나열 금지 |
| 항목별 LLM 프롬프트·스키마 | 04-llm-api-spec §7 | 참조만 |
| 항목별 화면 렌더링·카드 스펙 | 02-screens §5 | 참조만 |
| 반응형 breakpoint (1024px) | 02-screens §0 | 픽셀 값을 다시 쓰지 않는다 |
| 알림 채널 정책 | 01-PRD §6 | 참조만 |
| 디자인 토큰 | [design-tokens/*.json](design-system/design-tokens/) | DesignSystem.md §3에서 참조만. 값 재선언 금지 |
| localStorage 스키마 | 03-data-spec §5-1 | 참조만 |
| 인수 기준 | 05-acceptance | PRD/화면명세에 인수 조건 박지 않음 |
| 미해결 이슈·결정 대기 | 06-open-issues | 본문에 "TBD" 사유를 길게 쓰지 않고 이슈 ID 참조 |

### 1-1. 참조 포맷 (필수 통일)

- 다른 문서: `[01-PRD §4](01-PRD.md#4-기능-범위)`
- 같은 문서 내: `§5-3`
- 미해결 이슈: `[DEV-001](06-open-issues.md#dev-001)`
- **금지**: "PRD에 따르면…", "데이터 스펙에서…" 같은 자연어 참조 (링크 없는 언급)

### 1-2. SSOT 위반 판단 기준

다음 중 하나라도 해당하면 SSOT 위반이다:
- 같은 식별자(M1, W2 등)의 **정의 문장**이 두 문서 이상에 존재
- 같은 숫자(V1 7개, breakpoint 1024px 등)가 두 문서 이상에 등장
- 같은 표(데이터 필드 목록 등)가 두 문서 이상에 존재

발견 시 즉시 SSOT 위치 한 곳만 남기고 나머지는 참조로 변경한다.

## 2. 누적 금지 — 수정 시 행동 규칙

### 2-1. 작업 시작 시 필수 절차

수정 요청을 받으면 **즉시 본문을 고치지 않는다**. 다음 순서를 반드시 따른다:

1. 대상 문서 전체를 읽는다
2. 영향받는 다른 문서(SSOT 표 기준)를 식별한다
3. 수정 후 구조를 텍스트로 먼저 제안한다
4. 사용자 승인 후 작성한다

### 2-2. Rewrite 우선 정책

- 수정 요청 = **해당 섹션 전체 재작성**. 끝에 덧붙이지 않는다.
- 새 항목 추가 시, 기존 항목과 **통합 가능한지 먼저 검토**한다.
- 통합 가능하면 통합 후 재작성. 별도 추가는 통합 불가능을 입증한 후에만.

### 2-3. 절대 하지 말 것

- 기존 문장을 그대로 두고 뒤에 "추가로…", "또한…"으로 잇기
- "기존 정책 유지하면서 X 추가" 같은 누적형 수정
- 이전 버전 내용을 취소선 또는 주석으로 남기기 (`~~삭제됨~~`, `<!-- 이전: -->`)
- 변경된 부분만 "diff처럼" 표기하기 (본문은 항상 최종 상태)

---

## 3. 변경이력 분리

### 3-1. 본문 금지 표현

본문에 다음 표현을 **절대 쓰지 않는다**:

- 버전 표기: `v3.4 신규`, `v3.5 변경`, `v3.6 제거`
- 변경 동사: `(신규)`, `(변경)`, `(추가)`, `(제거)`, `(이전)`
- 취소선: `~~M6 위키 문서~~`
- 시점 언급: `이전에는…`, `지금까지는…`, `기존…`
- 주석 형태 이력: `<!-- v3.5에서 변경 -->`

### 3-2. 변경 이력은 어디에?

모든 변경은 **CHANGELOG.md** (또는 각 문서 상단의 frontmatter `version` 필드)에만 기록한다. 본문은 항상 "지금 이 순간의 진실"만 담는다.

### 3-3. 헤더에 버전 박지 않기

❌ `### 2-10. 기준 객체 컨텍스트 (PrimaryObjectContext) — v3.4 신규`
✅ `### 2-10. 기준 객체 컨텍스트 (PrimaryObjectContext)`

헤더에서 "신규", "변경" 같은 단어를 모두 제거한다.

## 4. 문서 역할 경계

각 문서는 자기 책임 범위만 다룬다. 경계를 넘는 내용은 해당 문서로 옮긴다.

| 문서 | 포함 | 제외 |
|---|---|---|
| 01-PRD | 비즈니스 의도, V1/V2 스코프, 기능 요구사항(What), 사용자 페르소나 | 화면 픽셀 값, API 스키마, 인수 조건, 데이터 필드 정의 |
| 02-screens | 화면 구조, UI 상태, 인터랙션, 컴포넌트 스펙 | 비즈니스 정책, 데이터 모델, LLM 프롬프트 |
| 03-data-spec | 데이터 모델, 필터, 변환 규칙, localStorage 스키마 | 화면 디자인, LLM 프롬프트 텍스트, 인수 기준 |
| 04-llm-api-spec | LLM 호출 계약, 프롬프트, 스키마, SSE 프로토콜 | 데이터 정의(03 참조), 화면 렌더링 결과 |
| 05-acceptance | Given-When-Then 인수 기준, 골든 패스 | 요구사항 설명, 구현 방법 |
| 06-open-issues | 결정 대기 항목, 결정 주체, 기한 | 결정된 사항(결정 후 본문 이동) |
| DesignSystem.md | 설계 원칙, 디자인 토큰(design-tokens/ 참조), 컴포넌트 분류 | 프로젝트별 구체 적용, 구현 세부 사항 |

### 4-1. 경계 위반 예시

- ❌ PRD에 "JSON Schema는 LLM이 다음과 같이 반환…" → 04로 이동
- ❌ 화면명세에 "이 데이터는 14일간 캐시…" → 03으로 이동
- ❌ 데이터 스펙에 "사용자는 X를 원할 것…" → 01로 이동

### 4-2. 경계 모호한 경우

- "정책"이 어디 들어가는지 모호하면: **그 정책의 적용 대상**으로 결정. 데이터 처리 정책 → 03, UI 동작 정책 → 02, 비즈니스 규칙 → 01.
- 같은 정보가 두 문서에 필요하면: SSOT 한 곳만 두고 다른 곳은 참조.

## 5. 길이 가드레일

길이는 누적의 결과 지표이지 그 자체가 문제는 아니다. 다음 순서로 대응한다.

1. 길어졌다고 느끼면 먼저 §1 SSOT 위반과 §3 변경이력 잔재를 점검한다.
   대부분의 비대화는 중복·이력 누적이 원인이다.
2. 중복·이력 정리 후에도 한 문서가 한 가지 주제를 다루기 어려울 만큼
   길어진 경우에만 분할을 검토한다.
3. 헤더 깊이는 `####`까지. 5단계(`#####`) 금지.
4. 한 표 30행 초과 시 분할 또는 별도 문서.

분할은 AI가 자동 제안하지 않는다. 사용자가 명시적으로 요청할 때만 분할안을 작성한다.

## 6. 자가 점검 체크리스트 (수정 완료 전 필수)

수정 작업 종료 전 다음을 모두 확인한다. 하나라도 해당하면 다시 작업한다.

### 6-1. SSOT 무결성

- [ ] 같은 정의가 두 문서 이상에 등장하는가?
- [ ] V1/V2 항목 수, 1024px breakpoint 같은 핵심 숫자가 모든 문서에서 동일한가?
- [ ] 다른 문서를 자연어로 언급한 곳에 링크가 있는가?

### 6-2. 변경이력 잔재

- [ ] 본문에 "v[숫자]", "신규", "변경", "추가", "제거", "기존" 단어가 남아 있는가?
- [ ] 헤더에 버전·변경 표기가 박혀 있는가?
- [ ] 취소선 또는 주석 형태로 이전 내용이 남아 있는가?

### 6-3. 구조

- [ ] 5단계 이상 헤더(`#####`)가 있는가?
- [ ] 1,200줄을 초과했는가?
- [ ] 동일 레벨에서 같은 주제의 섹션이 중복 생성됐는가?

### 6-4. 결과물

- [ ] 수정 후 문서가 수정 전보다 길어졌는가? (길어졌다면 통합 누락 의심)
- [ ] 변경된 내용이 SSOT 매핑 표(§1)에 영향을 주는가? 영향 시 다른 문서도 동기화했는가?

## 7. 자유 형식 글쓰기 금지

설명체 서술을 최소화한다. 가능하면 표·리스트·정형 블록으로 전환한다.

- ❌ "이 항목은 사용자가 오전에 일정을 확인할 수 있도록 도와주며, 일반적으로 8시에 발송되고…"
- ✅ 표로: `| 항목 | 발송 시간 | 목적 |`

"일반적으로", "보통", "아마도", "~인 경우가 많다" 같은 추측성 부사를 금지한다. 정해진 사실만 적는다. 결정되지 않았으면 06-open-issues에 등록하고 본문에는 `[06 OI-XXX]` 참조만 남긴다.

## 9. 작업 시작 시 AI에게 (프롬프트 템플릿)

문서 수정을 요청할 때 다음 템플릿을 사용한다:

```
대상 문서: [파일명 §섹션]
요청 사항: [무엇을 어떻게 바꿀지]

다음 규칙을 반드시 지켜라:
1. CLAUDE.md §1 SSOT 표를 먼저 확인하고 다른 문서에 영향이 있는지 보고할 것
2. 기존 내용을 누적하지 말고 해당 섹션 전체를 재작성할 것
3. 변경이력 표현(v3.4, 신규, 변경 등)을 본문에 쓰지 말 것
4. 작업 전에 수정 후 구조를 먼저 제안하고 승인받을 것
5. 수정 완료 후 §6 체크리스트를 점검 결과로 출력할 것
```
