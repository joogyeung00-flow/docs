## 1. P1 — M1~W5 항목 정의 분산 정리

### 현황

브리핑 항목(M1, M2, M3, M5, W1, W2, W5)의 정의가 다음 4개 문서에 모두 존재한다.

- 01-PRD §5-3, §5-5 — 항목 라벨, 설명, 데이터 요건
- 02-screens §5-2, §5-3, §5-4 — 카드별 라벨, 데이터 표시
- 03-data-spec §1-5, §3-1 ~ §3-8 — 식별자, 소스, 데이터 모델
- 04-llm-api-spec §3, §7 — 항목별 프롬프트 (라벨·설명 포함)

### 정리 작업

**SSOT 결정**: 03-data-spec §1-5 (그룹핑 표) + §3 (항목별 데이터 요구사항)

**작업 절차**:

1. 03-data-spec §3을 항목별 SSOT로 확정. 항목 식별자·라벨·소스·데이터 요건을 여기 한 곳에서만 정의.
2. 01-PRD §5-3, §5-5에서 항목 정의 제거. 대신 다음과 같이 변경:
   ```
   ### 5-3. 오전 브리핑
   설정 시간(기본 08:00)에 4개 항목을 발송한다.
   대상 항목: [03-data-spec §1-5 참조](03-data-spec.md#1-5-서비스별-브리핑-항목-그룹핑)
   ```
3. 02-screens §5에서 항목별 데이터 필드 설명 제거. 카드 UI 스펙(레이아웃, 컴포넌트)만 남김.
4. 04-llm-api-spec §7에서 라벨·설명 재서술 제거. 프롬프트 본문 외의 항목 메타데이터는 03 참조.

### 검증

- M1 항목의 "라벨" 정의가 한 문서에만 존재하는가?
- 항목 추가/제거 시 03만 수정해도 모든 문서 정합이 유지되는가?

## 2. P2 — V1 스코프 숫자 불일치 정리

### 현황

같은 V1 스코프의 항목 수가 문서마다 다르게 표기됨.

- 01-PRD: "오전 브리핑 4개 항목 / 퇴근 브리핑 3개 항목" → 합 7개
- 03-data-spec: "V1 정기 브리핑 7개 항목 (v3.6에서 8 → 7개)"
- 04-llm-api-spec §7: "**V1 8항목**" — M4 포함된 상태로 미수정

### 정리 작업

**SSOT 결정**: 01-PRD §4 (V1/V2 스코프)

**작업 절차**:

1. 01-PRD §4에서 V1 항목 수와 목록을 확정 (`V1 정기 브리핑: 7개 항목 (M1, M2, M3, M5, W1, W2, W5)`)
2. 다른 문서에서 "V1 7개", "V1 8항목" 같은 숫자 표기를 모두 제거
3. 다른 문서가 항목 수를 언급해야 할 경우: `V1 정기 브리핑 항목 ([01-PRD §4](01-PRD.md#4-기능-범위) 참조)` 형태로만
4. 04-llm-api-spec §7 헤더의 "V1 8항목"을 "V1 정기 브리핑 항목"으로 수정. M4 관련 하위 섹션(§7-4) 제거 또는 V2 별도 섹션으로 이동.

### 검증

- 패키지 전체에서 "7개", "8개", "8항목" 검색 시 01-PRD §4 외에는 결과 없음
- M4 관련 서술이 V2 영역에만 존재

## 3. P3 — 변경이력 본문 침투 정리

### 현황

본문에 박힌 변경 이력 흔적: 총 295건
- 01-PRD: 74건
- 02-screens: 42건
- 03-data-spec: 110건 (가장 심각)
- 04-llm-api-spec: 69건

대표 패턴:
- `### 2-10. 기준 객체 컨텍스트 (PrimaryObjectContext) — v3.4 신규`
- `> **위키의 위치 (v3.4 변경)**: …`
- `(BRIEFING.overdueTasks는 v3.6 이후 LLM에 전달되지 않음 — M4 제거.)`
- `~~M6 최근 업데이트된 위키 문서~~`

### 정리 작업

**작업 절차**:

1. **CHANGELOG.md 생성** — 각 문서별로 별도 또는 통합. 최소한 다음 형식:
   ```
   ## 03-data-spec.md
   - v3.6 (2026-04-XX): M4 V1 스코프 제거, 공휴일 옵트인 추가
   - v3.5 (2026-04-XX): ChatRoom 모델 추가, M5 schedule 타입 제거
   - v3.4: PrimaryObjectContext 추가, M6 제거
   ```

2. **본문 일괄 정리** — 다음 패턴을 자동/수동으로 모두 제거:
   - 헤더의 ` — v3.X 신규`, ` (v3.X 변경)` 등 → 제거
   - 본문 중 `(v3.X 신규)`, `(v3.X 변경)`, `(v3.X에서 X)` → 제거
   - 취소선 `~~…~~` → 완전 제거 (해당 내용 자체를 삭제)
   - `이전에는…`, `기존…`, `지금까지는…` 같은 시점 표현 → 현재 상태만 남기고 재서술

3. **frontmatter 정비** — 각 문서 상단에 현재 버전만 표기:
   ```yaml
   ---
   문서: 데이터 스펙
   feature-id: FLOW-001
   version: 3.6
   ---
   ```

### 우선순위

03-data-spec (110건)부터 시작. 가장 정합성에 영향이 큼.

### 검증

- 각 문서에서 정규식 `v[0-9]\.[0-9]`, `(신규|변경|추가|제거)`, `~~`, `이전에는|기존` 검색 시 결과 0건
- 헤더에 ` — ` 뒤 변경 표기 없음

## 4. P4 — 문서 역할 경계 위반 정리

### 현황

CLAUDE.md §4의 역할 경계를 위반한 항목들.

| 위치 | 내용 | 이동 대상 |
|---|---|---|
| 01-PRD §5-X | 내용 브리핑 공통 파이프라인 (기술 로직) | 04-llm-api-spec |
| 01-PRD §5-Y | AI 퀵 액션 생성 규칙 (생성 로직) | 04-llm-api-spec |
| 01-PRD §5-Z | AI 응답 생성 상태 관리 정책 (UI 상태) | 02-screens |
| 01-PRD §6 | 알림 채널 정책 (정책) | PRD에 유지하되 채널별 렌더링 스펙은 02로 이동 |
| 02-screens §0 | 반응형 정책 | 02 유지 (UI 정책이므로 OK) |

### 정리 작업

**작업 절차**:

1. 각 위반 섹션의 **내용은 이동, 위치는 참조 링크로**:
   ```
   ## 5-X. 내용 브리핑 공통 파이프라인
   본 파이프라인의 기술 명세는 [04-llm-api-spec §9](04-llm-api-spec.md#9-자유-대화-내용-브리핑-인터페이스)에 정의된다.
   비즈니스 의도: 사용자가 일정·업무·게시글에 대해 자유 대화로 깊이 질문할 수 있도록 한다.
   ```

2. PRD에는 **"왜(Why)"와 "무엇(What)"만**, 다른 문서로 **"어떻게(How)"** 이동.

3. 이동 시 두 문서 모두에 같은 내용이 존재하지 않도록 주의 (P1과 동일한 SSOT 원칙).

### 검증

- 01-PRD에서 "JSON", "Schema", "프롬프트", "스키마" 같은 기술 용어가 §4 이전 또는 의도 설명 외에 등장하지 않음
- 02-screens에 비즈니스 정책(Why)이 들어 있지 않음

## 5. P5 — 거대 문서 분할 검토 (P1~P4 완료 후)

### 원칙

분할은 정리 작업의 **결과**이지 별도 작업이 아니다. P1(항목 정의 분산), P3(변경이력 침투), P4(경계 위반)를 먼저 정리하면 상당량이 축소된다. 축소 후에도 한 문서가 단일 주제를 담기 어려울 만큼 길어진 경우에만 분할을 검토한다.

### 절차

1. P1~P4 정리 완료 후 각 문서 줄 수를 재측정한다.
2. 1,200줄 이하로 떨어진 문서는 분할하지 않는다.
3. 1,200줄을 여전히 초과하는 문서에 대해서만 분할 필요성을 판단한다. 판단 기준은 줄 수가 아니라 "한 문서가 두 개 이상의 독립 주제를 다루는가"이다.
4. 분할이 필요하다고 판단되면 사용자에게 분할안을 제안하고 승인 후 진행한다.

### 작업 시 주의

- 분할안을 미리 만들어 두지 않는다. P1~P4 정리 후 실제 상태를 보고 결정한다.
- 분할 시 SSOT 위반 위험이 커진다(같은 개념이 여러 파일에 흩어짐). 분할로 얻는 가독성이 SSOT 비용을 초과하는 경우에만 진행한다.
- 분할 진행 시 00-README.md의 문서 목록·읽는 순서 표를 갱신한다.
