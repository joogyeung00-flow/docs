# 01-PRD — 플로우 AI 첫 화면 개선

## 1. 개요

| 항목 | 내용 |
|---|---|
| 프로젝트명 | 플로우 AI 첫 화면 개선 |
| 목적 | 첫 화면에서 즐겨찾기 섹션을 제거하고, 입력 중심 구조를 강화하며, 입력창 하단 슬라이드형 배너와 기능별 예시 프롬프트를 통해 첫 질문 시작과 운영 메시지 전달 효율을 함께 높인다 |
| 작성 목적 | 홈 화면 개편의 목표, 범위, 사용자 경험, 운영 방향 정의 |

## 2. 배경

- 첫 화면은 중앙 입력창 중심의 질문 시작 경험을 제공하나, 하단 즐겨찾기 영역이 큰 면적을 차지해 핵심 과업(빠른 질문 시작)과의 연결성이 낮다.
- 서비스 공지·신규 기능 안내·활용 가이드를 노출할 전용 영역이 부족하다. 상단 공지 바만으로는 메시지 주목도와 클릭 유도가 제한적이다.
- 기능 칩은 존재하나, 각 기능으로 어떤 요청이 가능한지 직관적으로 이해시키는 보조 장치가 부족해 첫 입력 전환의 심리적 장벽이 발생한다.

## 3. 문제 정의

| 번호 | 문제 |
|---|---|
| P1 | 즐겨찾기 영역과 홈 핵심 목적의 연결성이 낮음 |
| P2 | 운영 메시지를 노출할 시각적 공간 부족 |
| P3 | 상단 공지 바만으로는 주요 공지·기능 안내 전달력 약함 |
| P4 | 기능 칩에 활용 예시가 없어 사용자가 입력을 망설임 |
| P5 | 입력창·즐겨찾기·공지 요소 간 시선 분산 |

## 4. 목표 / 비목표

### 4-1. 목표

- 첫 화면 핵심 행동을 `질문 입력`으로 선명하게 만든다.
- 즐겨찾기 제거로 확보한 공간에 슬라이드형 배너 영역을 신설한다.
- 공지·기능 소개·활용 가이드를 효과적으로 노출한다.
- 기능 선택 시 예시 프롬프트로 첫 질문 전환을 높인다.

### 4-2. 비목표

- AI 챗봇·AI 업무매니저·AI 비서·AI 헬퍼·AI 플레이북스의 기능 자체 개편
- 입력창 기능 전면 개편
- 즐겨찾기 기능 삭제
- 대화 내역 구조 변경
- 추천 알고리즘 고도화
- 즐겨찾기 이관 구조 상세 설계

## 5. 타겟 사용자

| 분류 | 특성 |
|---|---|
| 일반 사용자 | 첫 화면에서 바로 질문을 시작 |
| 신규/라이트 사용자 | 각 기능의 활용 방법을 빠르게 이해해야 함 |
| 운영 메시지 수신자 | 공지·업데이트·신규 기능 안내를 홈에서 확인 |

## 6. 핵심 가설

| 번호 | 가설 |
|---|---|
| H1 | 즐겨찾기를 첫 화면에서 제거하면 질문 시작 흐름이 단순해진다 |
| H2 | 입력창 하단 슬라이드형 배너는 공지·기능 안내 도달률·클릭률을 높인다 |
| H3 | 기능 선택 시 예시 프롬프트 제공은 활용 이해도와 첫 질문 입력 전환율을 높인다 |

## 7. 개선 방향

- 첫 화면에서 즐겨찾기 섹션을 제거한다.
- 입력 중심 구조를 유지한다.
- 입력창 하단 슬라이드형 배너를 도입한다.
- 기능 선택 시 예시 프롬프트를 동적으로 노출한다.
- 즐겨찾기는 첫 화면 외 별도 영역으로 이관하되, 상세 설계는 별도 기획에서 진행한다.

화면 구조·상태·인터랙션 명세는 [02-screens](02-screens.md) 참조.

## 8. 사용자 시나리오

```
1. 첫 화면 진입
2. 자유대화 상태에서 기능 칩과 입력창 확인
3. 입력형 기능칩 선택
4. 선택 기능에 맞는 예시 프롬프트 확인
5. 예시 프롬프트 클릭 또는 직접 입력으로 질문 시작
6. (필요 시) 입력창 하단 슬라이드형 배너에서 공지·신규 기능 안내·활용 가이드 확인
7. AI 챗봇 칩 클릭 시 챗봇 홈으로 이동
```

상태별 화면 동작은 [02-screens](02-screens.md) 참조.

## 9. 정책

### 9-1. 예시 프롬프트 정책

| 항목 | 정책 |
|---|---|
| 연결 대상 | 입력형 기능에만 연결 |
| 운영 방식 | 기능별 고정 세트 |
| 점검 기준 | 기능 목적, 이해도, 활용성 |

개수·노출 조건은 [02-screens §3](02-screens.md#3-기능별-예시-프롬프트) 참조.

### 9-2. 배너 노출 정책

| 항목 | 정책 |
|---|---|
| 노출 위치 | 입력창 하단 슬라이드형 배너 |
| 첫 장 노출 | 우선순위가 가장 높은 배너 |
| 최대 노출 개수 | 3개 권장 |
| 긴급 공지 | 상단 공지 바 우선 노출 원칙 |

배너 위치·자동 롤링 등 화면 스펙은 [02-screens §5](02-screens.md#5-슬라이드형-배너) 참조.

### 9-3. 배너 운영 정책 (백오피스)

배너는 백오피스에서 등록·수정·종료한다.

#### 9-3-1. 필수 입력값

| 필드 | 설명 |
|---|---|
| 제목 | 배너 헤드라인 |
| 설명 | 본문 메시지 |
| CTA | 행동 유도 버튼 텍스트 |
| 링크 또는 액션 | CTA 클릭 시 동작 |
| 노출 기간 | 시작·종료 시점 |
| 우선순위 | 노출 순서 결정 |

#### 9-3-2. 종료 처리

종료된 배너는 자동 비노출한다.

## 10. 성공 지표

| 분류 | 지표 |
|---|---|
| 입력 전환 | 첫 질문 입력 전환율, 홈 진입 후 평균 첫 액션 도달 시간 |
| 기능 이해 | 기능 선택 칩 클릭률, 예시 프롬프트 클릭률, 예시 프롬프트 클릭 후 전송 전환율 |
| 운영 메시지 | 배너 CTR, 공지/기능 소개 페이지 유입률 |
| 이탈/만족 | 첫 화면 이탈률, 즐겨찾기 제거 이후 관련 VOC 발생량 |

## 11. 기대 효과

- 첫 화면의 역할이 명확해진다.
- 사용자가 각 기능 활용 방법을 빠르게 이해한다.
- 빈 입력창에 대한 심리적 장벽이 낮아진다.
- 질문 시작 전환율이 높아진다.
- 운영 메시지·추천 콘텐츠 노출 효율이 향상된다.
- 즐겨찾기 카드로 인한 시선 분산이 감소한다.

## 12. 리스크 및 대응

| 리스크 | 대응 |
|---|---|
| 예시 프롬프트가 많아지면 선택 피로 발생 | 기능별 2~4개로 제한 ([02-screens §3-1](02-screens.md#3-1-노출-조건)) |
| 배너 강조로 질문 시작 흐름 방해 | 입력창 하단 배치, 고정 높이, 제한된 개수 운영 ([02-screens §5-1](02-screens.md#5-1-위치구조)) |
| 사용자가 즐겨찾기 미노출을 불편하게 인식 | 기능과 저장 데이터는 유지하고 별도 영역으로 이관 ([02-screens §6](02-screens.md#6-즐겨찾기)) |
| 공지 바와 배너 메시지 중복 | 공지 바는 짧은 중요 공지, 배너는 상세 안내로 역할 분리 ([02-screens §0-2](02-screens.md#0-2-화면-요소-역할-분리)) |

## 13. 결론

본 개선은 홈의 역할을 `저장 항목 소비 화면`에서 `질문 시작을 돕는 AI 홈`으로 재정의하는 작업이다. 즐겨찾기 제거, 입력 중심 구조 유지, 기능 선택 기반 예시 프롬프트, 입력창 하단 슬라이드형 배너를 하나의 세트로 설계한다.

미해결 사항은 [06-open-issues](06-open-issues.md) 참조.
