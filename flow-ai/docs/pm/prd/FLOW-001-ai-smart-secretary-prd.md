# PRD: AI 스마트 비서 (AI Smart Secretary)

---

## 1. 개요

### 1-1. 제품 비전

> "Flow에 쌓인 업무 맥락을 누구보다 잘 이해하고, 사용자가 묻기 전에 먼저 챙겨주는 AI 업무 파트너"

AI 스마트 비서는 단순한 챗봇이나 알림 봇이 아닙니다. 협업 도구 Flow에 누적된 메시지, 일정, 업무, 파일 데이터를 바탕으로 사용자의 업무 상태를 파악하고, **능동적(proactive)**으로 정보를 제공하며 다음 행동을 유도하는 엔터프라이즈 AI 비서입니다.

### 1-2. 배경 및 문제 정의

**현황 문제**

| 문제 | 구체적 현상 |
|------|------------|
| 정보 과부하 | 사용자가 출근 후 수십 개의 채널·프로젝트를 직접 순회하며 업무를 파악해야 함 |
| 누락 리스크 | 마감 임박 업무, 미회신 메시지를 사용자가 능동적으로 체크하지 않으면 놓칠 수 있음 |
| 맥락 단절 | 장기 부재(휴가, 출장) 후 복귀 시 업무 현황 파악에 과도한 시간 소요 |
| 도구 분산 | 별도 캘린더·리마인더 앱을 병행 사용해야 하는 불편함 |

**기회**

Flow는 이미 메시지, 업무(Task), 일정, 채널 멤버십 등 풍부한 업무 데이터를 보유하고 있습니다. 이 데이터를 AI로 해석해 사용자에게 선제적으로 전달하면, 별도 도구 없이도 업무 파악과 실행이 Flow 안에서 완결될 수 있습니다.

### 1-3. 목표

**비즈니스 목표**
- Flow AI 구독 전환율 향상: AI 스마트 비서를 AI 플랜의 핵심 가치로 포지셔닝
- MAU 대비 DAU 비율(Stickiness) 개선: 매일 방문할 이유 제공
- 기업 고객 갱신율 향상: 업무 효율 개선을 통한 ROI 증명

**사용자 목표**
- 출근 후 3분 이내 오늘 해야 할 일 파악 가능
- 중요한 업무 누락 없이 처리
- Flow 와 flow AI의 결합으로 업무 루틴 완결

## 2. 타겟 유저

### 2-1. 주요 사용자 페르소나

#### 페르소나 A: 바쁜 팀 리더 - "김과장" (35세, 마케팅팀 팀장)
- **상황**: 5~10명 팀 관리, 하루 평균 50건 이상 메시지 수신
- **고충**: 팀원 업무 진행 상황을 파악하느라 아침 시간을 소모, 자신의 업무는 뒷전
- **니즈**: 팀 전체 현황을 한눈에 파악하고 빠르게 의사결정하고 싶음
- **AI 비서 활용 포인트**: 아침 브리핑으로 팀 업무 요약, 미회신 메시지 우선순위 파악

#### 페르소나 B: 멀티태스킹 실무자 - "이대리" (28세, 개발팀 프론트엔드)
- **상황**: 3개 이상 프로젝트 동시 참여, 여러 채널에서 멘션 다수
- **고충**: 마감 관리 실패, 우선순위 혼란, 중요 메시지 놓침
- **니즈**: 지금 가장 급한 것을 알려주고, 잊지 않게 상기시켜줬으면 함
- **AI 비서 활용 포인트**: 지능형 리마인더, 마감 임박 알림

#### 페르소나 C: 복귀 직원 - "박주임" (32세, 인사팀)
- **상황**: 2주 육아휴직 후 복귀
- **고충**: 어디서부터 파악해야 할지 막막함, 중요한 변경사항을 놓칠 불안
- **니즈**: 내가 없는 동안 뭐가 바뀌었는지 빠르게 요약해줬으면 함
- **AI 비서 활용 포인트**: 복귀 브리핑 시나리오

### 2-2. 사용 시나리오

#### 시나리오 1: 일상적인 아침 출근
```
1. 이대리가 09:00 Push 알림 수신 (AI 비서 "플레인"의 아침 브리핑)
2. 알림 탭 → 브리핑 대화 세션 진입
3. 브리핑: 오늘 마감 업무 3건, 어제 미확인 멘션 5건, 팀원 부재 정보
4. 이대리가 마감 업무 항목의 [업무 브리핑] 버튼 탭 → AI가 해당 업무 내용 요약
5. 세션에서 바로 이어서 대화하며 처리 시작
```

#### 시나리오 2: 리마인더 수신 및 스누즈
```
1. 김과장이 마감 D-1 업무를 아직 미완료 상태로 오후 3시 경과
2. AI 비서가 리마인더 알림 발송: "내일이 마감인 'Q2 마케팅 보고서' 아직 미완료예요"
3. 김과장이 "2시간 후 다시 알림" 스누즈 선택
4. 오후 5시 재알림 수신, 김과장이 업무 처리
```

#### 시나리오 3: 복귀 브리핑
```
1. 박주임이 10일 휴가 후 복귀, Flow 앱 접속
2. AI 비서가 복귀 브리핑 대화 세션 자동 생성 + Push 알림 발송
3. 세션 진입 → AI 첫 메시지: 주요 결정사항 3건, 미처리 댓글 요약, 새 프로젝트 2건
4. 박주임이 핵심 내용만 빠르게 파악하고 업무 복귀
```

#### 시나리오 4: 브리핑 세션에서 이어서 대화
```
1. 이대리가 브리핑 세션의 "미확인 댓글 5건" 항목 확인
2. 세션 입력창에 "이 중 가장 급한 것만 알려줘" 자유 입력
3. AI가 브리핑 컨텍스트 기반으로 우선순위 분석 결과 응답
```

### V2 (향후 고려)

- 음성 브리핑 (TTS 기반)
- 브리핑 스케줄 사용자 커스터마이징 (시간대 직접 설정)
- 팀/그룹 브리핑 (팀장 전용 팀 현황 뷰)
- 외부 캘린더 연동 (Google Calendar, Outlook)

- 브리핑 이력 아카이브 조회

### 범위 외 (Out of Scope)

- AI 기반 업무 자동 완료·수정 (인간의 최종 확인 없는 자동 처리)
- Flow 외부 서비스 데이터 자동 수집 (별도 연동 설정 없이)
- 실시간 음성 대화 인터페이스
- AI가 사용자 대신 메시지 발송
- 타 사용자 데이터 열람 (개인정보 보호 원칙 준수)

## 4. 기능 요구사항

### 4-1. 온보딩

#### 개요
신규 사용자가 AI 스마트 비서를 처음 활성화할 때 진행하는 2단계 설정 흐름입니다. 목표 소요시간은 30초 이내입니다.

#### 유저 스토리
- 사용자는 AI 스마트 비서를 처음 시작할 때 비서의 이름을 직접 지을 수 있다.
- 사용자는 키워드를 선택하면 AI가 아바타 이미지를 생성해준다.
- 사용자는 온보딩 중 언제든 이전 단계로 돌아갈 수 있다.
- 사용자는 온보딩을 완료하지 않고 나갔다가 재진입 시 이전에 완료한 단계부터 이어서 진행할 수 있다.

#### 상세 요구사항

**단계 1: 비서 이름 설정**
- 입력 필드: 최대 10자 (한글, 영문, 숫자 허용)
- 기본값: "플레인" (서비스 기본 이름)
- 금지 패턴: 특수문자, 욕설·비속어 (금지어 필터 적용)
- 중복 불허: 시스템 내 예약어 사용 불가 (예: "Flow", "AI", "관리자")
- 유효성 검사: 입력 즉시 실시간 검사, 오류 시 인라인 메시지 표시

**단계 2: 아바타 생성**
- 방식: 서비스가 보유한 기본 프롬프트 + 사용자 키워드 선택 → AI 이미지 생성
- 키워드 카테고리 (예시, 확정 전 디자인 협의 필요):
  - 분위기: 차분한 / 활기찬 / 따뜻한 / 시크한
  - 스타일: 심플 / 귀여운 / 전문적인 / 위트있는
- 동작 흐름:
  1. 사용자가 각 카테고리에서 키워드 선택
  2. 선택된 키워드가 기본 프롬프트에 반영되어 이미지 생성 요청 발송
  3. 생성 중 로딩 인디케이터 표시 (예상 소요: 3~5초)
  4. 생성된 이미지 미리보기 → "확인" 또는 키워드 변경 후 재생성
- 기본 프롬프트: 서비스 내부 보관, 사용자에게 노출하지 않음
- 생성된 이미지는 서버에 저장하고 `avatar_url`로 참조

**온보딩 완료 처리**
- 완료 시 축하 애니메이션 + 첫 인사말 메시지 표시
- 온보딩 완료 상태를 서버에 저장 (재진입 시 온보딩 스킵)
- 완료 후 아침 브리핑 수신 안내 및 알림 권한 요청 (모바일)

#### 예외 케이스
| 상황 | 처리 방법 |
|------|----------|
| 금지어 입력 | 실시간 오류 메시지 표시, 다음 단계 진행 불가 |
| 10자 초과 입력 | 입력 자체를 10자에서 차단 (추가 입력 불가) |
| 네트워크 오류로 저장 실패 | 재시도 버튼 표시, 로컬 임시 저장 후 재시도 |
| 온보딩 중 앱 강제 종료 | 완료된 단계까지 서버 저장, 재접속 시 이어서 진행 |
| 비서 이름 미입력 상태로 다음 진행 시도 | 기본값("플레인") 자동 적용 후 진행 허용 |
| 아바타 이미지 생성 실패 (API 오류) | 기본 폴백 이미지 자동 적용, 설정에서 재생성 가능 안내 |
| 생성된 이미지 부적절 콘텐츠 감지 | 이미지 미표시, 자동 재생성 1회 시도 후 실패 시 기본 이미지 적용 |

### 4-2. 아침 브리핑

#### 개요
매일 아침 AI 비서가 사용자에게 자동으로 보내는 프로액티브 메시지입니다. 브리핑 발송 시 새로운 대화 세션이 생성되며, 사용자는 Push 알림을 탭하거나 대화 히스토리에서 해당 세션으로 진입할 수 있습니다. Flow의 업무·캘린더 데이터를 LLM이 해석하여 자연어 요약으로 생성하며, 4개 섹션으로 구성되고 각 섹션에는 동적 CTA가 포함됩니다.

#### Flow 데이터 소스

**업무 데이터**
| 필드 | 값 |
|------|----|
| 담당자 | 사용자 계정 |
| 시작일 / 마감일 | 날짜 |
| 상태 | 대기 / 진행 / 완료 / 보류 |
| 우선순위 | 낮음 / 보통 / 높음 / 긴급 |
| 업무 유형 | 상위업무 / 하위업무 |
| 상위업무명 | 텍스트 (하위업무인 경우에만 존재) |

**캘린더 데이터**
| 필드 | 값 |
|------|----|
| 일시 | 날짜 + 시간 |
| 참석자 | 사용자 계정 목록 |
| 장소 | 텍스트 (선택) |
| 내용 | 텍스트 (선택) |

**댓글 / 멘션 데이터**
| 필드 | 값 |
|------|----|
| 대상 게시물 유형 | 글 / 업무 / 일정 |
| 대상 게시물 제목 | 텍스트 |
| 작성자 | 사용자 계정 |
| 작성 시각 | 날짜 + 시간 |
| 댓글 유형 | 댓글 / 대댓글 구분 |
| 멘션 여부 | 나를 멘션했는지 여부 (boolean) |
| 내 게시물 여부 | 댓글이 달린 게시물의 작성자가 나인지 여부 (boolean) |
| 내 댓글 대댓글 여부 | 내가 작성한 댓글에 달린 대댓글인지 여부 (boolean) |
| 댓글 내용 | 텍스트 원문 (LLM 요약용으로 전달) |

**LLM에 전달하는 댓글 범위**
- 나를 멘션한 댓글·대댓글 (게시물 유형 무관)
- 내가 작성한 게시물(글·업무·일정)에 달린 댓글
- 내가 작성한 댓글에 달린 대댓글

> LLM 응답에는 댓글 원문을 그대로 노출하지 않고 요약된 형태로만 제공. Prompt Guard 마스킹 정책 적용.

#### 유저 스토리
- 사용자는 매일 아침 설정된 시간에 브리핑 대화 세션을 받을 수 있다.
- 사용자는 브리핑 세션에서 각 항목의 [내용 브리핑] 버튼으로 AI 요약을 추가로 받을 수 있다.
- 사용자는 브리핑 세션에서 바로 이어서 자유롭게 대화할 수 있다.
- 사용자는 브리핑이 필요 없는 날 수동으로 비활성화할 수 있다.

#### 브리핑 발송 정책
- **기본 발송 시간**: 평일 오전 8:00 (사용자 기기 타임존 기준)
- **세션 생성**: 발송 시각에 새 대화 세션 자동 생성, AI 브리핑 메시지가 첫 메시지로 기록됨
- **Push 알림**: 세션 생성과 동시에 발송 → 탭 시 해당 세션으로 이동
- **발송 조건**: 전날 또는 당일 Flow 접속 이력이 있는 사용자에게만 발송
- **미발송 조건**: 4개 섹션 모두 표시할 데이터가 없는 경우 세션 미생성, 알림 없음
- **캐시 정책**: 동일 사용자 동일 날짜 브리핑은 캐시 우선 제공 (cache key = 사용자 ID + 날짜 + 기능 타입)
- **캐시 무효화 조건**: 브리핑 발송 후 30분 이내 주요 데이터 변경(신규 긴급 업무 배정 등) 시 재생성 알림

#### 4개 섹션 상세 정의

**CTA 공통 동작 원칙**

각 항목에는 게시물 직접 이동 대신 **"내용 브리핑" 버튼**을 제공합니다. 버튼을 누르면 해당 항목과 관련된 정보를 AI가 수집·요약하여 대화창에 표시합니다.

```
사용자: [내용 브리핑] 버튼 탭
  └── AI가 해당 항목 기준으로 검색 수행
        ├── Flow 데이터 검색 (관련 업무·댓글·게시글)
        ├── Flow 위키 검색 (관련 문서)
        └── 웹 검색 (필요 시)
  └── 수집 결과를 종합하여 자연어 요약 응답
```

예) "오후 3시 스프린트 리뷰" 항목의 [내용 브리핑] 탭 →
> "스프린트 리뷰는 완료된 작업을 팀과 검토하는 자리예요. Flow에서 이번 스프린트 업무 12건 중 9건 완료, 3건 진행 중이에요. 위키에 '스프린트 리뷰 가이드'가 있어요."

---

**섹션 1: 오늘의 일정**
- 데이터 소스: Flow 캘린더 — 당일 일시 기준 이벤트
- LLM 역할: 일시·장소·참석자를 자연어 한 줄 요약으로 변환
  - 예) "오후 3시, 본사 3층 회의실에서 김팀장 외 3명과 스프린트 리뷰 예정"
- 표시 건수: 최대 3건 (초과 시 "+N개 더" 표시), 시작 시간 오름차순
- CTA: 각 일정 항목마다 [일정 브리핑] 버튼
- 빈 상태: "오늘 예정된 일정이 없어요"

**섹션 2: 마감 업무**
- 데이터 소스: Flow 업무 — 내가 담당자이고 마감일이 당일 또는 D+1인 업무 (상태: 대기·진행)
  - 상위업무·하위업무 모두 포함
- LLM 역할: 업무명·마감일·우선순위를 조합하여 우선순위 순으로 자연어 요약
  - 예) "오늘까지 '[긴급] 계약서 검토'를 완료해야 해요"
  - 예) "오늘까지 'PRD 작성 (스마트비서 기획)'을 완료해야 해요"
- 하위업무 표시 방식: 상위업무명 > 하위업무명 계층 표시
  - 예) 스마트비서 기획 > PRD 작성
- 우선순위 표시: 긴급(빨강), 높음(주황), 보통(기본) 배지
- 표시 건수: 최대 5건 (긴급 → 높음 → 보통 → 낮음 순 정렬)
- CTA: 각 업무 항목마다 [업무 브리핑] 버튼
- 빈 상태: "오늘과 내일 마감인 업무가 없어요"

**섹션 3: 부재중 변화**
- 데이터 소스: 전날 마지막 접속 이후 ~ 브리핑 생성 시각까지의 변경사항
  - 내가 담당자로 새로 배정된 업무 건수
  - 내 업무의 상태·우선순위 변경 건수
  - 나를 멘션한 댓글·대댓글 건수 (글·업무·일정 게시물 모두 포함)
  - 내가 작성한 게시물에 달린 댓글 건수
  - 내가 작성한 댓글에 달린 대댓글 건수
- LLM 역할: 변경 건수 및 댓글 내용을 종합하여 "놓친 것" 중심으로 자연어 요약
  - 예) "어젯밤 사이 업무 2건이 새로 배정됐고, 김팀장이 '기획안 작성'에 디자인 수정 방향에 대한 의견을 남겼어요"
- CTA: "전체 보기" → 미확인 항목 목록으로 이동
- 빈 상태: "놓친 변화가 없어요"

**섹션 4: 확인 필요**
- 데이터 소스: 아래 조건 중 하나 이상 해당하는 항목
  - Flow 업무: 마감일이 지났고 상태가 완료·보류가 아닌 업무 (담당자: 나, 상위업무·하위업무 모두 포함)
  - Flow 업무: 우선순위가 긴급이고 상태가 대기·진행인 업무 (담당자: 나, 상위업무·하위업무 모두 포함)
  - 하위업무 표시 방식: 상위업무명 > 하위업무명 계층 표시
  - 댓글·멘션: 나를 멘션한 댓글·대댓글, 내가 작성한 게시물에 달린 댓글, 내가 작성한 댓글에 달린 대댓글 (글·업무·일정 모두 포함, 당일 기준)
- LLM 역할: 항목별 맥락에 맞는 액션을 제안하는 자연어 요약
  - 예) "'견적서 제출'이 어제까지였는데 아직 완료 전이에요. 오늘 처리해볼까요?"
  - 예) "이지영님이 '기획안' 글에서 일정 조율을 요청했어요"
- CTA: 각 항목별 맥락 버튼 (예: "업무 확인", "상태 변경", "댓글 보기")
- 빈 상태: "지금 당장 급한 건 없어요, 여유롭게 시작해요!"

#### 세션 내 이어서 대화

브리핑 세션은 AI 첫 메시지 이후 사용자가 바로 이어서 대화할 수 있는 구조입니다.

```
[브리핑 세션]
  AI 첫 메시지: 4개 섹션 브리핑
  └── 사용자 입력 가능
        예) "마감 업무 중 가장 급한 거 먼저 알려줘"
            "어제 내가 없는 동안 팀에서 뭐가 결정됐어?"
      └── AI 응답 (브리핑 컨텍스트 유지)
```

- 브리핑 데이터는 해당 세션의 컨텍스트로 LLM에 자동 전달됨 (사용자에게 비표시)
- 대화 기록은 해당 세션의 채팅 히스토리에 누적 저장
- 대화 중 특정 업무·일정으로 딥링크 이동 가능

#### 예외 케이스
| 상황 | 처리 방법 |
|------|----------|
| LLM 생성 실패 | 오류 메시지 표시 + [다시 생성] 버튼 제공. 재시도 후에도 실패 시 "잠시 후 다시 시도해주세요" 안내 |
| 4개 섹션 모두 데이터 없음 | 브리핑 미발송. 알림 없음 |
| 주말·공휴일 | 브리핑 미발송 (기본값), 설정에서 발송 허용 가능 |
| 사용자가 브리핑 수신 거부 설정 | 발송 중단, 설정 화면에서 재활성화 가능 |
| 타임존 변경 (해외 출장 등) | 기기 타임존 변경 감지 시 다음날부터 새 타임존 적용 |

### 4-3. 지능형 리마인더

#### 개요
사용자가 설정하지 않아도 AI가 상황을 판단하여 자동으로 보내는 업무·일정 리마인더입니다. 리마인더 발송 시 새로운 대화 세션이 생성되며, Push 알림 탭 시 해당 세션으로 이동합니다.

#### 유저 스토리
- 사용자는 마감이 임박한 미완료 업무에 대해 자동 리마인더를 받을 수 있다.
- 사용자는 장시간 미회신 상태인 중요 메시지에 대해 리마인더를 받을 수 있다.
- 사용자는 리마인더를 스누즈하거나 해제할 수 있다.
- 사용자는 리마인더 수신 설정(활성/비활성, 민감도)을 조절할 수 있다.
- 사용자는 하루에 받을 리마인더 최대 횟수를 설정할 수 있다.

#### 트리거 타입

**타입 1: 마감 임박 리마인더**

| 조건 | 발송 타이밍 |
|------|------------|
| 마감 D-1 (내일 마감) | 당일 오후 2:00 |
| 마감 당일 | 오전 9:30 (브리핑 이후) |
| 마감 초과 (D+1 이상) | 익일 오전 10:00 |

- 대상: 내가 담당자인 업무 (상위업무·하위업무 모두 포함, 설정에서 범위 조정 가능)
- 조건: 업무 상태가 대기·진행
- 하위업무 알림 표시: 상위업무명 > 하위업무명 계층 표시

**타입 2: 일정 시작 전 리마인더**

| 조건 | 발송 타이밍 |
|------|------------|
| 일정 시작 30분 전 | 시작 시각 기준 30분 전 |

- 대상: 내가 참석자로 등록된 캘린더 일정
- 조건: 당일 일정에 한함 (익일 이후 일정은 아침 브리핑에서 안내)
- 알림 내용: 일정명, 시작 시각, 장소(있는 경우), 참석자 수

**타입 3: 미회신 리마인더**

| 조건 | 발송 타이밍 |
|------|------------|
| 채팅 미확인 24시간 이상 | 해당 시간 경과 후 익일 오전 |
| 일정 미참석 24시간 이상 | 해당 시간 경과 후 익일 오전 |

#### 알림 피로 방지 (신규)
- **동일 업무 중복 알림 방지**: 동일 업무에 대한 리마인더 최소 간격 4시간
- **저녁 시간 알림 제한**: 기본 오후 9:00 이후 알림 미발송 (Do Not Disturb)
- **주말 알림 설정**: 기본 비활성, 사용자가 주말 알림 허용 가능

#### 스누즈 규칙

| 스누즈 옵션 | 재발송 시점 |
|------------|------------|
| 1시간 후 | 현재 시각 + 1시간 |
| 2시간 후 | 현재 시각 + 2시간 |
| 내일 오전 | 익일 오전 9:00 |
| 해제 (이 업무 더 이상 알리지 않기) | 해당 업무 리마인더 완전 중단 |

- 스누즈 후 재발송은 일일 상한 횟수에 포함되지 않음 (사용자 선택 행동이므로)
- 업무 완료 처리 시 자동으로 모든 스누즈 취소

#### 리마인더 메시지 구조

```
[AI 비서 이름]  14:02

📌 마감 D-1 업무 알림
"Q2 마케팅 보고서"가 내일까지예요.
현재 업무 상태: 요청

[업무 바로가기]  [스누즈]  [완료 처리]
                    └── [1시간 후] [2시간 후] [내일 오전] [더 이상 알리지 않기]
```

#### 예외 케이스
| 상황 | 처리 방법 |
|------|----------|
| 업무가 이미 완료된 경우 발송 직전 확인 | 발송 취소 |
| 사용자가 일일 상한 초과 | 해당 리마인더 다음날로 이월 |
| 중복 트리거 동시 발생 | 가장 우선순위 높은 1건만 발송, 나머지 병합 또는 이월 |
| 리마인더 발송 중 네트워크 오류 | 지수 백오프로 최대 3회 재시도 후 실패 처리 |

### 4-4. 개인화 설정

#### 개요
사용자가 AI 비서의 외형, 호칭 방식, 커뮤니케이션 스타일을 커스터마이징할 수 있는 설정 화면입니다.

#### 유저 스토리
- 사용자는 온보딩 이후에도 AI 비서 이름, 아바타를 변경할 수 있다.
- 사용자는 AI 비서의 말투(톤)를 자신에게 맞게 선택할 수 있다.
- 사용자는 알림 수신 시간, 알림 횟수 상한 등 알림 정책을 설정할 수 있다.

#### 설정 항목 및 프롬프트 파라미터 매핑

| 설정 항목 | 옵션 | 프롬프트 파라미터 | 적용 예시 |
|----------|------|-----------------|---------|
| 말투 | 간결형 | `tone: concise` | "마감 3건. 확인 필요." |
| 말투 | 공손형 | `tone: formal` | "오늘 마감이 3건 있으십니다. 확인 부탁드립니다." |
| 말투 | 친근형 | `tone: friendly` | "오늘 마감 3건이나 있어! 같이 해결해보자!" |
| 비서 이름 | 텍스트 (최대 10자) | `assistant_name: "{name}"` | 시스템 프롬프트 서명에 적용 |
| 아바타 | 키워드 선택 → AI 생성 | `avatar_url: "{url}"` | UI 표시용 (프롬프트 미적용) |

> **[확장 예정]** 말투 설정은 현재 AI 스마트 비서 전용으로 적용됩니다. 향후 Flow AI 전체(AI 챗봇, AI 업무 매니저 등)의 공통 말투 설정으로 통합될 예정이며, 개발 시 글로벌 설정으로 확장 가능한 구조로 구현을 권장합니다.

#### 알림 설정 항목

| 설정 항목 | 기본값 | 범위 |
|----------|--------|------|
| 아침 브리핑 수신 | ON | ON/OFF |
| 브리핑 발송 시간 | 오전 09:00 | 오전 06:00 ~ 오전 11:00 |
| 마감 임박 리마인더 수신 | ON | ON/OFF |
| 일정 시작 전 리마인더 수신 | ON | ON/OFF |
| 일정 시작 전 알림 시간 | 30분 전 | 10분 / 30분 / 1시간 |
| 미회신 리마인더 수신 | ON | ON/OFF |
| 일일 리마인더 상한 | 5회 | 3~10회 |
| 야간 알림 제한 시작 시간 | 오후 09:00 | 오후 07:00 ~ 오후 11:00 |
| 주말 알림 허용 | OFF | ON/OFF |

#### 설정 변경 즉시 반영 정책
- 말투 설정은 다음 스마트비서 응답부터 즉시 반영
- 알림 시간 설정은 다음 발송 사이클부터 반영
- 변경 이력은 서버에 저장 (감사 로그 목적)

### 4-5. 알림 채널 정책

#### 개요
AI 비서 알림이 모바일, 데스크톱, 웹 환경에서 각각 어떻게 렌더링되는지 정의합니다.

#### 채널별 렌더링 스펙

**모바일 (iOS / Android)**

| 구분 | 스펙 |
|------|------|
| 아침 브리핑 | Push 알림 (핵심 요약) → 탭 시 브리핑 대화 세션으로 이동 |
| 마감 임박 리마인더 | Push 알림 (제목 + 1줄 요약) → 탭 시 리마인더 대화 세션으로 이동 |
| 일정 시작 전 리마인더 | Push 알림 (일정명 + 시작 시각 + 장소) → 탭 시 리마인더 대화 세션으로 이동 |
| 미회신 리마인더 | Push 알림 (제목 + 1줄 요약) → 탭 시 리마인더 대화 세션으로 이동 |
| Push 알림 타이틀 | "[비서 이름] AI 비서" |
| Push 알림 본문 | 최대 2줄 (약 100자), 핵심 내용만 |
| 알림 아이콘 | 비서 아바타 이미지 (AI 생성 이미지, 미생성 시 기본 폴백 이미지) |
| 알림 액션 버튼 | 최대 2개 (주요 CTA + "닫기") |
| Deep Link | 알림 탭 시 해당 브리핑/리마인더 대화 세션으로 바로 이동 |
| 야간 알림 | 기기 DND 설정 우선 적용 |

**PC (WEB)**

| 구분 | 스펙 |
|------|------|
| 아침 브리핑 | 브라우저 Web Push (권한 허용 시) → 클릭 시 브리핑 대화 세션으로 이동 |
| 리마인더 | 브라우저 Web Push → 클릭 시 리마인더 대화 세션으로 이동 |
| 알림 권한 미허용 시 | Flow 웹앱 내 상단 인앱 배너로만 표시 → 배너 클릭 시 해당 세션으로 이동 |
| 브라우저 미지원 환경 | 인앱 알림 배지 + AI 비서 대화 히스토리에서 확인 |

#### 알림 우선순위 처리
- 동일 시점에 복수 알림 발생 시: 브리핑 > 일정 시작 전 리마인더 > 마감 초과 리마인더 > 미회신 리마인더 순
- 묶음 알림: 동시 발생 3건 이상 시 "오늘 확인할 항목 N건이 있어요" 통합 알림으로 발송
- 일정 시작 전 리마인더는 묶음 알림 대상에서 제외 (시간 민감도 높음, 단건 발송)

## 5. 비기능 요구사항

### 5-3. 보안 및 개인정보

**Prompt Guard**
- 자동 마스킹 대상: 이름, 이메일, 전화번호, 주민등록번호, 신용카드 번호
- 금지어 차단: 시스템 프롬프트 탈취 시도, 역할극 공격, 탈옥 패턴 차단
- 처리 방식: LLM 전달 전 마스킹 → 응답 후 원복

**데이터 격리**
- 사용자 데이터는 소속 조직 범위 내에서만 접근 가능
- 타 사용자 데이터 절대 열람 불가 (쿼리 레이어에서 사용자 ID 필터 강제 적용)
- AI 생성 브리핑에 타인의 개인 데이터가 포함되지 않도록 필터링

**로그 및 감사**
- 모든 LLM 요청·응답 로그 90일 보관
- 개인정보 마스킹 후 로그 저장
- GDPR / 개인정보보호법 준수 (사용자 데이터 삭제 요청 처리 7일 이내)

## 7. UX 원칙 및 톤앤매너

### UX 원칙

| 원칙 | 설명 |
|------|------|
| 선제적(Proactive) | 사용자가 묻기 전에 필요한 정보를 먼저 제공 |
| 간결함(Conciseness) | 핵심만 전달, 불필요한 장황함 제거 |
| 실행 가능성(Actionability) | 모든 알림에는 즉시 실행 가능한 CTA 포함 |
| 투명성(Transparency) | AI가 어떤 데이터를 바탕으로 판단했는지 이해 가능하게 표시 |
| 부담 없음(Low Friction) | 사용자에게 추가 설정이나 입력을 최소화 |

### 톤앤매너 기본값

- 기본 말투: 공손형 (호칭 + 존댓말)
- 문장 길이: 1~2문장으로 핵심 전달
- 부정적 표현 지양: "못 했어요" 대신 "아직 진행 중이에요"
- 사용자 이름 호칭: Flow AI 내 정보의 "사용자 이름" 필드 값 사용

### AI 비서 페르소나

- 이름: 기본 "플레인" (flow+brain, 사용자 변경 가능)
- 성격: 꼼꼼하고 따뜻함, 업무에 진심, 사용자의 편의를 최우선
- 금지 표현: 과도한 감탄사, 지나친 격려 ("대박이에요!", "너무 잘하셨어요!" 등)
- 권장 표현: "확인해보시면 좋을 것 같아요", "아직 처리 전인 항목이 있어요"
